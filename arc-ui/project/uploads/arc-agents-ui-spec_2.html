Arc Agents · Frontend Design Brief
A dashboard that lives with you
through the day
This spec describes the visual and interaction design for the Arc Agents frontend — a single app that surfaces outputs from all 28 agents across a 6-tab navigation. The core design principle: the app is not a settings panel. It's a living briefing that changes state as the day progresses — morning looks different from evening, Monday looks different from Sunday. Agents push cards; the user pulls details. Notifications surface at the right moment, not all at once.

Implementation note for developer
The frontend does not run agents — it reads their cached output. Agents write to Notion (and optionally a local JSON store). The app fetches from Notion API or a Flask endpoint and renders. Refresh happens on app open + every 15 minutes in background.

Design note
Schedule strip scrolls horizontally to show the full day. Current time is marked with a live red line — blocks in the past dim slightly. Tapping any block expands it to show agent detail, options to reschedule, or mark done. The strip never disappears — it's always accessible by scrolling down past the feed cards.
Interaction states on feed cards
Tap card
Expands inline — shows full agent output text without leaving the tab. A "View in Notion" link at bottom for full detail.
Swipe left
Dismiss / done — card fades out with a subtle green flash. Logged as "acted on" for Mirror's feedback learning.
Swipe right
Snooze 2 hours — card moves to bottom of feed with a small clock badge. Good for reminders and nudges.
Long press
Options sheet — "Ignore always", "Remind tomorrow", "Send to agent" (opens chat with that agent), "View history".
Pull to refresh
Re-fetch from Notion — re-runs the frontend data pull. Shows last-updated timestamp below the nav bar.

Critical design note
Zara's output is visually rich — outfit suggestions need large image tiles (not a list). Three outfit cards side-by-side, each showing the Notion-stored clothing photo, the occasion, and the weather context. A "wear today" tap marks it in the log. This is the most image-heavy view in the app — give it breathing room and white space.

Agent detail view — tap any agent to open
Tap agent
Full-screen slide-up panel. Shows: agent name + role description, last run timestamp and token cost, today's full output (formatted), run history (last 7 days), and a "Chat with [Agent]" button — opens a text input that sends a message to that specific agent's system prompt context.
Chat button
"Chat with Aria" — opens a minimal chat interface where your message is appended to Aria's system prompt and context. Results write back to Notion. This is how you ask Aria to dig deeper on a story, or ask Nora to modify tonight's dinner.
Run now button
Manual trigger — re-runs that agent on demand. Shows a spinner, then displays fresh output. Useful after adding new data to Arc.

Design principle
Council tab should use more white space, larger serif typography — it is a reflective space. On non-Sunday weekdays, it shows a small "Next Council: Sunday 8pm" card and a rolling Mirror insight. It should never feel cluttered.
Push Notifications
Time-aware notification system
Notifications are not blasts — they are timed to the exact moment each agent's output is relevant. Echo (Twilio WhatsApp) is the primary delivery for urgent reminders; in-app push handles context-rich cards.

Notification philosophy
Never more than 2 in-app notifications per hour. If multiple agents trigger in the same window, batch them into one card ("3 agents have updates"). Echo WhatsApp messages are separate — these are the high-priority channel. In-app notifications are ambient; WhatsApp is urgent/important.

Essential requirements — the brief
RequirementSpec
Platform targetMobile-first PWA or React Native. Must work well as a phone home screen app. Desktop secondary.
NavigationBottom tab bar · 6 tabs · Today / Health / Career / Life / Agents / Council · icons + labels
Data sourceReads from Notion API (agent outputs) + optional local Flask JSON endpoint · no backend of its own · refresh every 15 min + on open
State modelApp knows current time + reads Arc cycle data (day of cycle) → this drives which cards surface at top of Today
Push notifications10 scheduled daily notifications · times are configurable · badge count on Today tab for unread cards
OfflineCache last Notion fetch · app must be usable offline with last-known data · show "last updated" timestamp
Chat interfaceEach agent has a simple chat modal · message → hits Flask endpoint → Claude API · response displayed inline · no threading needed
FontsCormorant Garamond (Google Fonts) + DM Mono + DM Sans — must load via CDN, not system fonts
AnimationsCard entrance: fade up · Tab switch: fade · Expand/collapse: smooth height transition · Notification: slide in from top · Nothing that feels heavy or gamified
Dark modeOptional phase 2 — not required for MVP. If added, invert the warm parchment palette to a deep warm charcoal, not pure black.
MVP scope — what to build first
Phase 1 MVP
Today tab only · reads Notion · morning brief card · schedule strip · 3 push notifications (morning, evening wind-down, one reminder) · static layout
Phase 2 — full app
All 6 tabs · time-aware card ordering · swipe gestures · sub-tab navigation within Health and Career · full notification schedule · agent chat modals
Phase 3 — polish
Cycle ring animation · Agents status grid · Mirror behaviour graph · Council transcript reader · Zara image tiles · Finn donut chart · Barcelona animations
Technology recommendation
Recommended stack for builder
React Native + Expo (easiest to go from web → phone app) — or a PWA in React if they're more comfortable with web. Either works. Key libraries: react-query (data fetching + caching), react-native-push-notification or expo-notifications, @notionhq/client for Notion reads. The backend (Flask) is already being built as part of Ch.8 of the learning plan — the frontend just calls its /api/today, /api/agents/[name] etc. endpoints.
