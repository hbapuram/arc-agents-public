Core Concept
Multi-source RAG — live data, conversational layer

Chapter 1 used hardcoded context. Chapter 2 introduces dynamic multi-source context from Apple Health and Notion, plus a conversational interface layer so each agent can answer follow-up questions with full personal context pre-loaded.

The fundamental shift from Ch1
From static context to live personal data
Chapter 1 agents read from a static JSON file — tasks, goals, a fixed list of commitments. Useful, but brittle. Chapter 2 agents pull from live sources that change daily: your Apple Watch metrics, Notion databases you update naturally, and a new Personal Care Cupboard that mirrors your physical shelf.

The key skill you're practising is selective extraction — reading a rich data source and pulling only the fields this specific agent needs. Nora never sees your sleep data. Dusk never touches your pantry. Felix never reads your skincare shelf. Lean context keeps prompts fast, cheap, and precise. Each agent builds a build_context() function that is responsible for this filtering step — it's the most important function in each agent file.

Wrist temperature delta from Apple Watch Series 8+ adds a second cycle phase signal. Wrist temperature rises ~0.3°C after ovulation and stays elevated through the luteal phase. get_cycle_phase() cross-references the Apple Health cycle day with wrist temp delta to give Flora, Nora, Luna, Cara, and Aurora a more confident phase signal than cycle day alone.
New in Ch2 — the conversational layer
Dedicated chat interfaces, not a shared chatbot
Each agent that benefits from back-and-forth dialogue gets its own dedicated chat interface — a separate HTML file pre-loaded with that agent's full personal context. You open nora_chat.html and Nora already knows your pantry, your cycle phase, your protein shortfall from this week, and your active dietary goals. You can ask "what should I make with what I have?" or "is this recipe OK for the luteal phase?" without any setup.

This is architecturally different from a shared chatbot. The system prompt is generated fresh on each page load by calling build_context() — so it always reflects today's data. The chat interface makes one API call per user message, using claude-haiku-3-5 for speed. Memory within a session is handled by passing the full conversation history in each request. Persistent memory across sessions — your preferences, things you told Nora last week — lives in a Notion "Preferences" property on your profile page, and is injected into the system prompt on load.

Agents with conversational interfaces: Nora (food advisor), Felix (fitness advisor), Cara (personal care advisor), Eve (evening coach), and Vera (Quick Compare — the single entry point for on-the-go queries).
New in Ch2 — Vera, the Quick Compare entry point
One chat, smart routing, short decisive verdicts
When you're in a supermarket deciding between two products, or in a pharmacy unsure whether a supplement suits you, you don't want to open the right agent, set up context, and explain yourself. You want to ask once and get a clear yes/no/which-one.

Vera is a single entry point that auto-routes your query to the right domain based on what you ask. If Vera isn't sure of the domain, she shows context chips: Food · Skin · Supplements · Fitness — you tap one and she re-runs with the right depth of context. No setup required.

New — Personal Care Cupboard schema
Six categories, timing-aware, expiry and price intelligence
Create a new Notion database called Personal Care Cupboard. Properties:

Name (title) · Category (select: Skincare / Body Care / Hair Care / Supplements / Oral Care / Tools) · Brand (text) · Status (select: Active / Low / Empty / Considering) · Timing (select: AM / PM / Both / Flexible / As-needed) · Medicated (checkbox) · Notes (text — sequencing rules, layering notes, doctor's instructions, reactions) · Cycle Phases (multi-select: Menstrual / Follicular / Ovulatory / Luteal)

Expiry (date) — the label expiry date printed on the product. · Opened (date) — the date you first opened it. Set this when you unseal a new product. · PAO (number, months) — Period After Opening. The small open-jar symbol on most skincare (e.g. 12M = use within 12 months of opening). Cara calculates effective expiry as min(Expiry label, Opened + PAO) — whichever comes first. · Price Paid (number, €) — what you paid. Combined with Volume this gives cost-per-use. · Volume (text) — "50ml", "30 tablets", "100g". Keeps repurchase decisions informed. · Rating (select: 1 / 2 / 3 / 4 / 5) — your satisfaction score. When a product hits Empty, Cara uses Rating ≥ 4 → add to repurchase list; Rating ≤ 2 → suggest a category alternative instead.

The "Considering" status is new: products with Status = Considering are things you're thinking of buying. Vera uses these when you ask "should I get this product?" — she compares it against what you already have in the same category (price per ml, overlapping function, whether you'd finish the Considering product before existing stock expires).

The Medicated flag and Notes field handle sequencing. The Expiry + PAO fields handle urgency. The Price + Volume + Rating fields handle value. All of this intelligence lives in your Notion data — Cara just reads, reasons, and presents.

PANTRY_DB  = os.getenv("NOTION_PANTRY_DB_ID")

def get_pantry(category=None):
    """Active pantry items (In Stock + Low) as list of dicts."""
    filt = {"or": [{"property": "Status", "select": {"equals": s}} for s in ("In Stock", "Low")]}
    if category:
        filt = {"and": [filt, {"property": "Category", "select": {"equals": category}}]}
    results = notion.databases.query(database_id=PANTRY_DB, filter=filt)["results"]
    items = []
    for r in results:
        p = r["properties"]
        items.append({
            "name":     p["Name"]["title"][0]["text"]["content"] if p["Name"]["title"] else "",
            "category": p["Category"]["select"]["name"] if p["Category"]["select"] else "",
            "quantity": p["Quantity"]["number"],
            "unit":     p["Unit"]["select"]["name"] if p["Unit"]["select"] else "",
            "status":   p["Status"]["select"]["name"] if p["Status"]["select"] else "",
            "expiry":   p["Expiry"]["date"]["start"] if p.get("Expiry", {}).get("date") else None,
            "protein":  p["Protein"]["number"] if p.get("Protein", {}).get("number") else None,
            "notes":    p["Notes"]["rich_text"][0]["text"]["content"] if p["Notes"]["rich_text"] else "",
        })
    return items


def get_habits_today():
    """All active habits + whether each was completed today."""
    filt    = {"property": "Active", "checkbox": {"equals": True}}
    results = notion.databases.query(database_id=HABITS_DB, filter=filt)["results"]
    today   = str(date.today())
    habits  = []
    for r in results:
        p    = r["properties"]
        last = p["Last Completed"]["date"]["start"] if p.get("Last Completed", {}).get("date") else None
        habits.append({
            "habit":           p["Habit"]["title"][0]["text"]["content"] if p["Habit"]["title"] else "",
            "category":        p["Category"]["select"]["name"] if p["Category"]["select"] else "",
            "frequency":       p["Frequency"]["select"]["name"] if p["Frequency"]["select"] else "Daily",
            "streak":          p["Streak"]["number"] if p.get("Streak", {}).get("number") else 0,
            "completed_today": last == today,
            "last_completed":  last,
        })
    return habits


def get_habit_streaks():
    """Lightweight streak-only version for Lyra's weekly review."""
    return [{"habit": h["habit"], "streak": h["streak"], "frequency": h["frequency"]}
            for h in get_habits_today()]


def get_hub_today():
    """Today's Hub page content as plain text — Aurora calls this to synthesise all agents."""
    blocks = notion.blocks.children.list(block_id=HUB_PAGE)["results"]
    lines  = []
    for b in blocks:
        btype = b["type"]
        if btype in ("heading_2", "heading_3"):
            text = "".join(t["plain_text"] for t in b[btype]["rich_text"])
            lines.append(f"## {text}")
        elif btype == "paragraph":
            text = "".join(t["plain_text"] for t in b["paragraph"]["rich_text"])
            if text.strip(): lines.append(text)
        elif btype == "bulleted_list_item":
            text = "".join(t["plain_text"] for t in b["bulleted_list_item"]["rich_text"])
            if text.strip(): lines.append(f"• {text}")
    return "\n".join(lines)

from datetime import datetime
from notion_client import Client
from dotenv import load_dotenv

load_dotenv()
notion   = Client(auth=os.getenv("NOTION_TOKEN"))
HUB_PAGE = os.getenv("NOTION_HUB_PAGE_ID")


def _text_to_blocks(content: str) -> list:
    """Map structured agent output (##, •, -, plain lines) to Notion block objects."""
    blocks = []
    for line in content.strip().split("\n"):
        line = line.strip()
        if not line:
            blocks.append({"object": "block", "type": "paragraph", "paragraph": {"rich_text": []}})
        elif line.startswith("## "):
            blocks.append({"object": "block", "type": "heading_3",
                           "heading_3": {"rich_text": [{"type": "text", "text": {"content": line[3:]}}]}})
        elif line.startswith("• ") or line.startswith("- "):
            blocks.append({"object": "block", "type": "bulleted_list_item",
                           "bulleted_list_item": {"rich_text": [{"type": "text", "text": {"content": line[2:]}}]}})
        else:
            blocks.append({"object": "block", "type": "paragraph",
                           "paragraph": {"rich_text": [{"type": "text", "text": {"content": line}}]}})
    return blocks


def write_to_hub(section_title: str, content: str):
    """Append a dated section to the Hub page. Every Ch2 agent calls this once per run."""
    today   = datetime.now().strftime("%Y-%m-%d")
    heading = f"{section_title} — {today}"
    blocks  = [
        {"object": "block", "type": "heading_2",
         "heading_2": {"rich_text": [{"type": "text", "text": {"content": heading}}]}},
        *_text_to_blocks(content),
        {"object": "block", "type": "divider", "divider": {}},
    ]
    # Notion caps at 100 blocks per request
    for i in range(0, len(blocks), 100):
        notion.blocks.children.append(block_id=HUB_PAGE, children=blocks[i:i+100])

This replaces Ch1's update_hub_section(block_id, content) pattern. Ch2 agents append new dated sections to the Hub rather than rewriting fixed block IDs — history is preserved. Test by calling write_to_hub("🧪 Test", "Line one\n• bullet\n• bullet") and checking your Hub page in Notion.

def _get_sheet():
    creds = service_account.Credentials.from_service_account_info(
        json.loads(os.getenv("FINCH_SERVICE_ACCOUNT_JSON")), scopes=SCOPES
    )
    return build("sheets", "v4", credentials=creds).spreadsheets()


def get_transactions(days: int = 30) -> list[dict]:
    """Last N days of Finch transactions as list of dicts."""
    result = _get_sheet().values().get(spreadsheetId=SHEET_ID, range="Sheet1!A:E").execute()
    rows   = result.get("values", [])
    if not rows: return []
    headers = rows[0]
    data    = [dict(zip(headers, r)) for r in rows[1:]]
    cutoff  = (datetime.today() - timedelta(days=days)).strftime("%Y-%m-%d")
    return [d for d in data if d.get("date", "") >= cutoff]

import os
from datetime import date, datetime, timedelta
from notion_client import Client

notion = Client(auth=os.getenv("NOTION_TOKEN"))
CARE_DB = os.getenv("NOTION_CARE_DB_ID")

def _effective_expiry(label_expiry: str, opened: str, pao_months: int) -> date | None:
    """Return the earlier of label expiry or opened+PAO. None if no data."""
    dates = []
    if label_expiry:
        dates.append(datetime.strptime(label_expiry, "%Y-%m-%d").date())
    if opened and pao_months:
        pao_date = datetime.strptime(opened, "%Y-%m-%d").date()
        # Approximate PAO: add months by replacing month, handling year rollover
        m = pao_date.month - 1 + pao_months
        pao_date = pao_date.replace(year=pao_date.year + m // 12, month=m % 12 + 1)
        dates.append(pao_date)
    return min(dates) if dates else None

def _expiry_flag(exp: date | None) -> str:
    if not exp: return "unknown"
    days = (exp - date.today()).days
    if days < 0:   return "expired"
    if days <= 30:  return "expires_soon"
    if days <= 90:  return "expires_3mo"
    return "ok"

            "expiry_flag": _expiry_flag(eff_exp),
            "days_left":   (eff_exp - date.today()).days if eff_exp else None,
            "price":       price,
            "volume":      p["Volume"]["rich_text"][0]["text"]["content"] if p.get("Volume", {}).get("rich_text") else None,
            "rating":      int(p["Rating"]["select"]["name"]) if p.get("Rating", {}).get("select") else None,
        })
    # Sort: expired first (flag them), then expires_soon, then by days_left ascending
    items.sort(key=lambda x: (
        x["expiry_flag"] not in ("expired", "expires_soon"),
        x["days_left"] if x["days_left"] is not None else 9999
    ))
    return items

def get_active_products(include_considering=False):
    """Return items grouped by category, sorted by expiry urgency within each group."""
    items = get_care_items(include_considering=include_considering)
    grouped = {}
    for item in items:
        grouped.setdefault(item["category"], []).append(item)
    return grouped

def get_repurchase_candidates():

import os, anthropic
from utils.care_reader import get_active_products, get_repurchase_candidates
from utils.apple_health_reader import get_cycle_phase
from utils.notion_writer import write_to_hub

client = anthropic.Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

def _fmt_product(p: dict) -> str:
    """Format a product for the system prompt, including expiry and price context."""
    base = p["name"]
    flag = p.get("expiry_flag", "ok")
    days = p.get("days_left")
    price = p.get("price")
    rating = p.get("rating")
    notes = p.get("notes", "")
    parts = [base]
    if flag == "expired":
        parts.append("[EXPIRED — discard or use at own risk]")
    elif flag == "expires_soon" and days is not None:
        worth_it = (rating or 3) >= 4
        action = "prioritise — use first" if worth_it else "use up then discontinue"
        price_note = f" (€{price} paid)" if price else ""
        parts.append(f"[expires in {days}d{price_note} — {action}]")
    elif flag == "expires_3mo" and days is not None:
        parts.append(f"[{days}d left]")
    if notes:
        parts.append(f"({notes})")
    return " — ".join(parts)

def build_context():
    products = get_active_products()    # dict keyed by category, sorted by expiry urgency
    phase = get_cycle_phase()
    phase_name = phase["name"]

    skincare = products.get("Skincare", [])
    # Split by timing; expiry-urgency sort is already applied by get_active_products()
    am_skin = [p for p in skincare if p["timing"] in ("AM", "Both")]
    pm_skin = [p for p in skincare if p["timing"] in ("PM", "Both")]
    medicated = [p for p in skincare if p["medicated"]]

    supps = products.get("Supplements", [])

    # Phase-restricted items
    restricted = [p for p in skincare if p["phases"] and phase_name not in p["phases"]]

    # Low stock alert
    low = [p["name"] for cat in products.values() for p in cat if p["status"] == "Low"]

    # Expiry alerts — expired or expiring within 30 days, across all categories
    expiry_alerts = [
        p for cat in products.values() for p in cat
        if p["expiry_flag"] in ("expired", "expires_soon")
    ]

    # Repurchase candidates (empty, rating ≥ 4)
    repurchase = get_repurchase_candidates()

    return {
        "am_skincare": am_skin, "pm_skincare": pm_skin,
        "medicated": medicated, "supplements": supps,
        "restricted_today": restricted, "low_stock": low,
        "expiry_alerts": expiry_alerts, "repurchase": repurchase,
        "phase": phase_name, "phase_day": phase["day"],
    }

def build_system_prompt(ctx):
    expiry_lines = [_fmt_product(p) for p in ctx["expiry_alerts"]] or ["None"]
    repurchase_lines = [
        f"{r['name']} ({r['category']}, rated {r['rating']}/5"
        + (f", €{r['price']}" if r["price"] else "") + ")"
        for r in ctx["repurchase"]
    ] or ["None"]
    return f"""You are Cara, a personal care advisor. You know this person's exact product shelf and its expiry status.

Current phase: {ctx['phase']} (day {ctx['phase_day']})
AM skincare (expiry-urgency order — use expiring items first):
{chr(10).join('  • ' + _fmt_product(p) for p in ctx['am_skincare'])}
PM skincare (expiry-urgency order, medicated last):
{chr(10).join('  • ' + _fmt_product(p) for p in ctx['pm_skincare'])}
Supplements: {[f"{p['name']} ({p['timing']})" for p in ctx['supplements']]}
Phase-restricted today: {[p['name'] for p in ctx['restricted_today']]}
Low stock: {ctx['low_stock']}
⚠️ Expiry alerts: {expiry_lines}
🛒 Repurchase list (empty, loved): {repurchase_lines}

Produce: (1) AM routine in expiry-urgency order with any expiry notes inline, (2) PM routine same, (3) supplement timing, (4) expiry action list if any alerts exist — one line per product with days left, price paid, and recommendation (use now / use up and replace / discard), (5) phase skin alert if relevant. Be specific. Under 400 words."""

def run():
    ctx = build_context()
    sys_prompt = build_system_prompt(ctx)
    response = client.messages.create(
        model="claude-haiku-3-5-20251001",
        max_tokens=500,
        system=sys_prompt,
        messages=[{"role": "user", "content": "Give me today's care routine."}]
    )
    write_to_hub("💆 Cara — Personal Care", response.content[0].text)

if __name__ == "__main__": run()

def detect_domain(query: str) -> str:
    q = query.lower()
    scores = {
        "food": sum(1 for k in FOOD_KEYWORDS if k in q),
        "skin": sum(1 for k in SKIN_KEYWORDS if k in q),
        "supplements": sum(1 for k in SUPP_KEYWORDS if k in q),
        "fitness": sum(1 for k in FITNESS_KEYWORDS if k in q),
    }
    best = max(scores, key=scores.get)
    return best if scores[best] > 0 else "ambiguous"

Goal Layer
Monthly → weekly → daily habit cascade

Goals give your agents purpose beyond daily data. Without them, Sage gives you a to-do list. With them, Sage gives you a day that moves you toward something specific.

The cascade structure
Three levels, one Notion database
All goals live in a single Notion Goals DB, differentiated by the Timeframe property (Monthly / Weekly / Daily). Monthly goals are the anchors — big directional targets for the month: "Hit 120g protein/day average", "Complete 3 strength sessions/week", "Keep skincare routine consistent for 28 days."

Weekly goals are decompositions of monthly ones, linked via the Parent Goal relation field: "Prep protein sources on Sunday", "Gym Monday/Wednesday/Friday", "Do PM skincare before 10pm each night." These are the ones Sage, Luna, and Eve actively reference each day.

Daily habits in the Habits DB are the mechanics — the specific behaviours that fulfil weekly goals. Lyra's job is to keep this chain coherent: if your weekly goals aren't being hit, she looks at which daily habits are slipping and proposes changes at the habit level rather than just repeating the goal.
Which agents read goals
Sage — reads weekly goals to prioritise the day's tasks against them
Luna — reads weekly goals to give context to the wellness check ("you're 2 gym sessions away from your weekly target")
Eve — reads weekly goals to build tomorrow's time-block plan around them
Finn — reads Finance category goals to contextualise spending
Aurora — reads weekly goals to frame the morning brief
Lyra — reads all goals for the weekly review and cascade check
Vera — reads active goals to inform Quick Compare verdicts ("you're in a protein-focus week, this product has 28g — yes")
Setting up your goal structure
Start with 2–3 monthly goals max — too many dilutes every agent's ability to prioritise
Write weekly goals as specific and measurable: "3 gym sessions" not "exercise more"
Link weekly goals to monthly using the Parent Goal relation — this is how Lyra checks the cascade
Keep Active checkbox up to date — agents filter to Active = true only
Lyra will propose archiving goals that haven't been touched for 3+ weeks — surface area creep is real
Cycle phase affects goal feasibility — Flora and Cara both read goals to give phase-calibrated expectations ("energy may be low this week — your strength goal is realistic but be flexible on intensity")
Feedback Loop
Mark suggestions as Used, Skipped, or Modified

Without a feedback loop, your agents are talking into a void. The Feedback Log closes the circuit — it lets you signal what's actually working, and lets Lyra surface patterns you might not notice yourself.

How the feedback loop works
Lightweight logging, pattern-aware review
When any agent makes a specific suggestion and you act (or don't), you log it in the Feedback Log DB. The logging is fast — you can create a Notion button on the Hub page that opens a pre-filled form with today's date and Agent pre-selected. The key fields are just: which agent, what suggestion (copy 3–5 words), and the status (Used / Skipped / Modified).

You don't need to log everything — just the suggestions that feel significant or that you consciously chose to ignore. Even 2–3 entries per week gives Lyra enough signal.

Lyra reads the Feedback Log during the Sunday review and looks for:

SCOPES = ["https://www.googleapis.com/auth/spreadsheets.readonly"]
SHEET_ID = os.getenv("ARC_FITNESS_SHEET_ID")

def _get_sheet():
    creds = service_account.Credentials.from_service_account_info(
        json.loads(os.getenv("FITNESS_SERVICE_ACCOUNT_JSON")), scopes=SCOPES
    )
    return build("sheets", "v4", credentials=creds).spreadsheets()

def get_health_log(days: int = 7) -> list[dict]:
    """Return last N days of health data as list of dicts."""
    sheet = _get_sheet()
    result = sheet.values().get(spreadsheetId=SHEET_ID, range="Sheet1!A:N").execute()
    rows = result.get("values", [])
    headers = rows[0]
    data = [dict(zip(headers, r)) for r in rows[1:]]
    return data[-days:]

def get_today() -> dict:
    return get_health_log(1)[-1]

def get_cycle_phase() -> dict:
    today = get_today()
    cycle_day = int(today.get("cycle_day", 1))
    wrist_delta = float(today.get("wrist_temp_delta", 0))

    if cycle_day <= 5:   phase_key, phase_name = "menstrual", "Menstrual"
    elif cycle_day <= 13: phase_key, phase_name = "follicular", "Follicular"
    elif cycle_day <= 17: phase_key, phase_name = "ovulatory", "Ovulatory"
    else:                 phase_key, phase_name = "luteal", "Luteal"

    # Wrist temp delta confirms luteal (post-ovulation) if > +0.2°C
    confidence = "confirmed" if (phase_key == "luteal" and wrist_delta > 0.2) \
                 or (phase_key != "luteal" and wrist_delta <= 0.2) \
                 else "estimated"

    energy_map = {"menstrual": "low", "follicular": "rising", "ovulatory": "peak", "luteal": "declining"}
    return {"key": phase_key, "name": phase_name, "day": cycle_day,
            "energy": energy_map[phase_key], "confidence": confidence,
            "wrist_temp_delta": wrist_delta}

Smart Decisions
Budget Pace Context — Spend-Aware Recommendations

Agents make different recommendations based on whether you're on-budget, overspending, or underspending this week. This prevents well-meaning suggestions that conflict with your financial reality.

Budget pace calculation
Weekly actual spend ÷ Weekly budget target
Example: Monthly grocery budget = €200. Weekly target = €200 ÷ 4.3 weeks ≈ €47.
This week's spend = €56. Pace = €56 ÷ €47 = 1.19 (19% overspend).

Pace 0.8: Underspending. Agents prioritize quality recommendations.
Pace 1.0: On target. Balanced approach — quality if cost difference is small.
Pace 1.2+: Overspending. Agents prioritize cheapest options and cost-cutting suggestions.

Flora's decision tree (simplified)

IF budget_pace > 1.15: Recommend cheapest store, flag budget pressure
ELIF budget_pace < 0.85: Recommend quality-preferred store if user has rated it highly
ELSE: Balanced — quality if cost difference is marginal (<10%)

Engineering
Production Patterns for Multi-Agent Systems

Patterns that ensure your 9-agent system is robust, fast, and maintainable in production.

1. Selective Context (600–800 chars per agent)

Don't pass entire databases to agents. Extract top 3–5 relevant entries. Example: Nora needs recent pantry expiration dates, not the entire pantry inventory. Keep context dict under 800 chars when stringified. Faster API calls, lower costs, clearer outputs.

2. Graceful Degradation

If Finch API is down, Vera still routes to Flora with cached budget data. If a reader utility fails, agent continues with what it has. No reader is essential — each context layer is optional. If all fail, agents still work with empty context.

3. Structured Parsing with Markers

Use markers in system prompts to delimit agent output: [DECISION_START] ... [DECISION_END]. More reliable than fragile regex parsing. Claude respects markers consistently.

4. Circuit Breaker for Failing APIs

If an external API fails 5 times in 60 seconds, stop calling it temporarily. Return cached data or skip that context layer. Prevents cascading failures when Finch, Notion, or Apple Health is slow.

5. Logging for Observability

Log every agent invocation: [VERA] Query: "where to buy spinach?" [VERA] Routed to: Flora [VERA] Context ready: cycle_phase=follicular, budget_pace=1.1 [FLORA] Response: "M&S, fresh, within budget". Helps debug when things go wrong.

Apple Watch Pipeline
iOS Shortcut → Apps Script → arc-fitness Sheet

This pipeline is the foundation for all health agents. If this stops working, all health agents degrade to using yesterday's cached data. Build it carefully and test it manually before relying on the cron schedule.

1
Create the arc-fitness Google Sheet
New Sheet with columns: date, steps, hrv, resting_hr, calories, sleep_duration, sleep_score, deep_sleep_min, rem_sleep_min, cycle_day, wrist_temp_delta, active_calories, stand_hours, vo2_max. Create a dedicated Google Cloud service account (separate from Finch) and share the Sheet with it (Editor). Store credentials as FITNESS_SERVICE_ACCOUNT_JSON in your .env.
2
Build the Apps Script webhook
In Google Apps Script, create a doPost(e) function that parses the incoming JSON and appends a row to the Sheet. Deploy as a Web App (Anyone can access — no auth needed since the URL is secret). Copy the deployment URL. The Apps Script runs as your account and has direct Sheet access — no service account needed here.
3
Build the iOS Shortcut
Create an Automation that runs at 11pm daily. Use Get Health Samples actions to read: Steps (today), HRV (latest), Resting Heart Rate, Active Energy, Sleep analysis, Wrist Temperature (if Series 8+), Cycle Tracking (cycle day). Compose a JSON dictionary and use Get Contents of URL (POST method, JSON body) to send to your Apps Script URL. Test by running the shortcut manually and checking the Sheet.
4
Schedule your morning run
Cron the morning run for 7am daily: 0 7 * * * cd /path/to/arc && python run_all.py morning. This gives the 11pm iOS Shortcut 8 hours to write the data before agents read it. Pulse check: 0 8,13,18 * * * python run_all.py pulse. Eve evening run: 0 19 * * * python run_all.py eve. Lyra Sunday: 0 20 * * 0 python run_all.py weekly.
5
Add graceful fallbacks
If today's row is missing (Shortcut failed to run), get_today() returns the most recent row and sets a data_fresh=False flag. Agents that read this flag include a note in their output: "(using yesterday's health data — today's metrics not yet available)". Aurora surfaces this in the morning brief so you know the data is stale.
